Webhook & Payment Trigger Disable Guide

Operations guide for disabling inbound payment webhooks, Clerk webhooks, payment cron triggers, and Clerk audit schedulers in event-site-manager-service without affecting core events, profiles, manual payments, promotion email batch jobs, or WhatsApp.

Backend only · Defaults: all flags false · Last updated: June 2026

1. Executive summary

Disabled by default (config flags in application.yml): payment provider inbound webhooks, Givebutter webhooks, Clerk inbound webhooks, payment-related cron triggers, and Clerk webhook audit scheduled jobs.

Still works unchanged: events/tickets/profiles CRUD, manual payments, promotion email and other non-payment batch jobs, WhatsApp/Twilio webhooks (JWT-protected), and client-side Clerk profile reconciliation on sign-in.

Design choice: feature flags, not code removal

Handlers remain registered but return HTTP 503 when disabled. This is reversible, keeps Spring wiring intact, and avoids merge conflicts from commented-out controllers.

3. Why each listener existed

  • Payment webhooks — Async updates from Stripe/PayPal/etc. after checkout; update user_payment_transaction and optionally subscriptions.
  • Givebutter — Donation events → ticket generation via PaymentOrchestrationService.processGivebutterWebhook.
  • Clerk webhooks — User/session lifecycle and audit table population. Primary profile reconciliation today is client-side on sign-in (see project cursor rules).
  • Payment cron triggers — HTTP entry points for AWS EventBridge/cron to forward Stripe subscription and fee reconciliation jobs to the batch-jobs microservice.
  • Clerk scheduled audit — Cleanup and retry for clerk_webhook_event rows when Clerk webhooks were active.

4. Disable checklist

Application configuration (defaults already off)
application:
  webhooks:
    payment-inbound:
      enabled: ${WEBHOOKS_PAYMENT_INBOUND_ENABLED:false}
    clerk-inbound:
      enabled: ${WEBHOOKS_CLERK_INBOUND_ENABLED:false}
    clerk-scheduled-tasks:
      enabled: ${WEBHOOKS_CLERK_SCHEDULED_ENABLED:false}
  batch-jobs:
    payment-cron:
      enabled: ${BATCH_JOBS_PAYMENT_CRON_ENABLED:false}
  webhook:
    ticket-generation:
      enabled: ${WEBHOOK_TICKET_GENERATION_ENABLED:false}

Leave env vars unset or set to false in production when no payment integration is active.

External systems

  • Stripe / Givebutter dashboards — Remove or disable webhook endpoints pointing at this API.
  • Clerk dashboard — Disable webhook endpoint (client-side reconciliation is primary).
  • AWS EventBridge / cron — Disable schedules calling POST /api/cron/subscription-renewal, stripe-fees-tax-update, stripe-ticket-batch-refund, manual-payment-summary.
  • Secrets — Optional cleanup: CLERK_WEBHOOK_SECRET, PAYMENT_ENCRYPTION_KEY, Stripe webhook secrets (script-only).

5. Re-enable guide (future payment integration)

  1. Set env vars to true (or override in application-prod.yml):
    • WEBHOOKS_PAYMENT_INBOUND_ENABLED=true
    • WEBHOOKS_CLERK_INBOUND_ENABLED=true (only if using Clerk webhooks again)
    • WEBHOOKS_CLERK_SCHEDULED_ENABLED=true (with Clerk webhooks)
    • BATCH_JOBS_PAYMENT_CRON_ENABLED=true
    • WEBHOOK_TICKET_GENERATION_ENABLED=true (only if tickets should be created from webhooks)
  2. Re-register webhook URLs in Stripe, Givebutter, and Clerk dashboards.
  3. Re-enable EventBridge/cron schedules for payment batch jobs.
  4. Run verification curls below — expect 200/202 instead of 503.

6. Verification

Replace BASE_URL with your backend (e.g. http://localhost:8080). With flags off, guarded endpoints return 503 with JSON like {"error":"... are disabled","enabled":"false"}.

Clerk webhook (expect 503)

Payment cron (expect 503)

Core APIs (expect unchanged behavior)
  • GET /api/event-details — 200 with auth/JWT as configured
  • Manual payment create/update — unchanged
  • POST /api/promotion-email-templates/{id}/send-bulk — unchanged (not guarded)
  • WhatsApp integration tests — WhatsAppWebhookResourceIT should still pass

Logs after 24h (flags off)

No messages from WebhookAuditServiceImpl scheduled cleanup or retry when clerk-scheduled-tasks.enabled=false.

7. Anti-pattern: do not comment out controllers

Why config flags beat commenting code
  • Commenting @PostMapping or @Scheduled breaks Spring wiring, tests, and OpenAPI docs unpredictably.
  • Re-enabling requires code changes and redeploy instead of env var flips.
  • Guard implementation: InboundWebhookGuard.java centralizes flag checks and 503 responses.
