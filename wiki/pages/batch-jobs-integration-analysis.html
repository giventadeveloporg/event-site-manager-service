Batch Jobs Integration Analysis

How event-site-manager-service (main backend, port 8080) integrates with event-site-manager-batch-jobs (batch microservice, port 8081, context path /batch-jobs). Generated from source analysis of both repositories.

Repositories: F:\project_workspace\event-site-manager-service · F:\project_workspace\event-site-manager-batch-jobs · Last reviewed: June 2026

Executive summary: Integration is HTTP REST only via Spring WebClient. There is no Maven dependency between the projects and no AWS Batch SDK usage in production paths (the software.amazon.awssdk:batch artifact in the main pom.xml is legacy/unused). Both services share the same PostgreSQL database; batch-jobs runs Spring Batch jobs and @Async tasks, then may call back into the main API with a service JWT.

1. Architecture

Integration model
  • Transport: Synchronous HTTP from main → batch-jobs; batch work runs asynchronously inside batch-jobs.
  • Client: BatchJobsWebClientConfiguration.java → WebClient bean batchJobsWebClient.
  • Base URL: batch-jobs.service.url (default http://localhost:8081/batch-jobs).
  • Outbound auth (triggers): Service JWT from BatchJobServiceAuthTokenProvider.java via POST {BATCH_JOBS_AUTH_BASE_URL}/api/authenticate.
  • Admin monitoring auth: Caller’s Authorization header is forwarded unchanged to batch-jobs.
  • Cron triggers: /api/cron/** accepts cron secret Bearer or normal JWT.
  • Inbound to batch-jobs: REST under /api/batch-jobs/* (see batch-jobs security configuration for production hardening).
  • Callbacks batch → main: BackendApiService in batch-jobs uses JWT to PATCH/GET main API resources (e.g. ticket transactions).

2. Features wired from main service

End-to-end: user/admin/cron → main API → batch-jobs

Eleven batch-jobs trigger endpoints are called from Java in the main service. Donation and some competition jobs exist only in batch-jobs (typically frontend-direct).

3. Data flows by feature

Promotion emails
Admin / frontend
  → POST /api/promotion-email-templates/{id}/send-test | send-bulk | send-to-subscribed
  → PromotionEmailServiceImpl
  → PromotionTestEmailBatchJobService OR BatchJobEmailService
  → POST batch-jobs /promotion-test-email OR /email
  → SES send, promotion_email_sent_log, batch_job_execution_log
Contact form
POST /api/contact-form-email/send
  → ContactFormServiceImpl → ContactFormBatchJobServiceImpl
  → POST /api/batch-jobs/contact-form-email
  → ContactFormEmailJobService (@Async), tenant FROM/COPY-TO, SES
Manual payments
POST /api/manual-payments/{id}/send-confirmation-email
  → ManualPaymentBatchJobService → POST /manual-payment-confirmation-email

Admin sets status RECEIVED → QR generated in main service
  → ManualPaymentRequestServiceImpl.triggerTicketEmailJob
  → POST /manual-payment-ticket-email
Cron / scheduled operations
EventBridge / external scheduler OR batch-jobs internal BatchJobScheduler
  → POST /api/cron/* (optional Bearer cron.secret)
  → BatchJobService → corresponding POST /api/batch-jobs/*

Stripe fees job may PATCH ticket transactions on main via batch-jobs BackendApiService.

9. Summary checklist

  • Main service is an orchestration and admin proxy; batch-jobs is the worker microservice.
  • 11 of 14 batch trigger REST paths are used from main-service Java; donation endpoints are batch-jobs/frontend only.
  • Shared PostgreSQL; execution history lives in batch-jobs DB tables.
  • Ensure same JWT secret across services when enabling auth on batch-jobs monitoring.
  • Set BATCH_JOBS_SERVICE_ENABLED=false to disable outbound calls without removing code paths.
