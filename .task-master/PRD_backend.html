Domain-Agnostic Payment Refactor – Backend PRD

Repository: E:\project_workspace\malayalees-us-site-boot
Goal: Implement a payment orchestration layer that supports multi-tenant Stripe (PaymentIntents/Checkout), optional ACP, and additional gateways (PayPal, Zeffy, Zelle manual, Revolut, CeFi charity) while centralizing fee settlement.

1. Architectural Overview

  • Payment Orchestration Service: Entry point for all payment operations (initialize, confirm, refund, status) via REST endpoints.
  • Provider Adapters: Implement PaymentService for each gateway (Stripe, PayPal, Zeffy, ZelleManual, Revolut, future CeFi). Responsible for provider API calls and translating results to our domain objects.
  • Configuration Layer: payment_provider_config table stores tenant-level provider credentials, feature flags (ACP, Zeffy, Zelle), and fallback ordering.
  • Transactions: payment_transaction, payment_transaction_item, payment_webhook_log, and new settlement/invoice tables provide the system of record.

2. Scope

Must Deliver

  • Stripe adapter refactor (PaymentIntent creation, Checkout Sessions, Link/Wallet support, webhook handling) with multi-tenant configuration.
  • Payment initialization endpoint (POST /api/payments/initialize) that returns provider-neutral session details.
  • Support for PayPal (Orders API), Zeffy (donation API/embeds), Zelle manual flows (pending transactions), Revolut (card acquiring), and CeFi charity flows.
  • Platform settlement job summarizing transactions per tenant/provider and generating invoices if fees aren’t auto-deducted.
  • ACP readiness: implement ACP endpoints (CreateCheckout, Update, Complete, Cancel) that wrap Stripe Instant Checkout once access is granted.

Out of Scope

  • UI changes (handled in frontend PRD).
  • Finance portal UI beyond API endpoints for settlements/invoices.
  • Automated reconciliation with accounting software.

3. API Surface

Endpoint Description Notes
POST /api/payments/initialize Create payment session. Returns PaymentSessionResponse with provider hint, clientSecret/sessionUrl, publishable key (if required). Derive schema from Swagger; support payment types: TICKET_SALE, DONATION, DONATION_CEFI, OFFERING, MEMBERSHIP_SUBSCRIPTION.
GET /api/payments/{transactionId} Fetch status for polling (used by frontend for Revolut/Zelle pending flows). Expose payment method details, settlement info, failure reasons.
POST /api/payments/{transactionId}/refund Issue full/partial refund through respective adapter. Log refund metadata in payment_transaction.
POST /api/payments/providers/stripe/instant-checkout Create Stripe Instant Checkout (ACP) session if tenant enabled. Returns hosted session URL; fallback to standard PaymentIntent if feature disabled.
POST /api/payments/zeffy/initialize Optional helper that returns Zeffy campaign info/embedded URL. Used when event flagged as zero-fee fundraiser.
POST /api/payments/zelle/manual Create pending transaction for Zelle instructions. Stores payer email, expected amount, memo instructions.

4. Database Updates

Tables

  • payment_transaction – add settlement_batch_id, platform_invoice_id, manual_payment_reference.

  • platform_invoice – invoices for outstanding platform fees; link to settlements.
  • membership_plan, membership_subscription – subscription definitions and user enrollments.

Migrations

  1. Create new tables with indexes described in master PRD.
  2. Migrate legacy event_ticket_transaction data into payment_transaction (Stripe only) for historical reference.
  3. Backfill payment_provider_config for existing tenants with Stripe configs.

5. Provider Adapters

  • StripePaymentAdapter: PaymentIntent, Checkout Session, Instant Checkout (ACP) creation, wallet support, subscription support (Stripe Billing), webhooks.
  • PayPalPaymentAdapter: Orders API, capture, refund, webhook validation.
  • ZeffyPaymentAdapter: Donation API integration; fallback instructions if unavailable.
  • ZelleManualAdapter: No external API; create pending transactions and expose admin confirmation endpoints.
  • RevolutPaymentAdapter: Create card sessions, handle redirect-based SCA, ingest settlement webhooks.
  • CeFiCharityAdapter: Integrate partner ACH/transfer provider for CeFi events (fetch settlement files).

All adapters must implement PaymentService methods (initialize, confirm, cancel, refund, webhook handling, supportedMethods).

6. Webhook Processing

  • Stripe: /api/webhooks/stripe – handle payment_intent.succeeded, payment_intent.payment_failed, checkout.session.completed, Instant Checkout events, refund events.
  • PayPal: /api/webhooks/paypal – capture/void notifications.
  • Zeffy: Import payout CSV or webhook if provided.
  • Revolut: Listen for payment_succeeded, payment_failed, settlement_created.

All webhook payloads must be logged in payment_webhook_log and processed idempotently.

7. Settlements & Fees

  1. Nightly job aggregates transactions → create platform_settlement entries with gross, processing fee, platform fee, net.
  2. If provider supports application fee (Stripe Connect), mark platform fee as collected; else generate platform_invoice.
  3. Expose admin API GET /api/finance/settlements and GET /api/finance/invoices for reporting dashboards.
  4. Provide POST /api/finance/invoices/{id}/charge to trigger auto-debit (Stripe Billing/ACH).

8. ACP Readiness Plan

  • Implement ACP REST endpoints according to Stripe spec (Create/Update/Complete/Cancel checkout).
  • Wrap existing Stripe PaymentIntent creation inside CompleteCheckout handler using Shared Payment Token.
  • Gate endpoints behind supports_acp flag in payment_provider_config.
  • Document onboarding steps (contact Stripe support, enable Instant Checkout, capture configuration IDs) – see master PRD section 15.11.

9. Security & Compliance

  • Store provider credentials encrypted (AES-256-GCM). Rotate keys annually.
  • Ensure webhook endpoints verify signatures and rate-limit requests.
  • Log all admin actions related to settlements/invoices.
  • Maintain PCI scope at SAQ A: backend never stores raw card data.

10. Testing Strategy

  • Unit tests for each adapter (mock provider APIs).
  • Integration tests for /api/payments/initialize covering all providers.
  • Contract tests using Swagger schema to ensure frontend/backends remain in sync.
  • End-to-end tests for Stripe and PayPal flows in staging (using Stripe test mode, PayPal sandbox).
  • Settlement job test to validate fee calculations and invoice generation.

11. Timeline (Backend)

  • Week 3: PayPal adapter, Zeffy integration, Zelle manual adapter.
  • Week 4: Revolut adapter, CeFi charity adapter, webhook handling hardening.

12. Acceptance Criteria

  • All payments initiated via backend endpoints; no provider secrets in frontend.
  • Adapters return consistent PaymentSession structure and handle webhooks idempotently.
  • Fee calculations recorded per transaction and summarized in settlement batches.
  • ACP endpoints functional behind feature flag and pass Stripe Instant Checkout certification.
  • Swagger documentation updated for every new endpoint and response.
