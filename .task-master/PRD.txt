1. EXECUTIVE SUMMARY
================================================================================

This document outlines the requirements for centralizing Clerk authentication
in the Spring Boot backend to serve multiple frontend applications (Next.js)
using a single set of Clerk API credentials.

⚠️ IMPORTANT: PROJECT CONTEXT
-----------------------------
This integration is being built ON TOP OF AN EXISTING, FULLY-FUNCTIONAL codebase:

✅ ALREADY IN PLACE:
- Spring Boot 3.x backend application (JHipster-based)
- PostgreSQL database with complete schema
- AWS Fargate deployment infrastructure
- user_profile table with tenant support
- JWT authentication framework
- Security configuration
- Multi-tenant architecture foundation
- CI/CD pipelines
- Development, staging, and production environments

🎯 WHAT WE'RE ADDING:
- Clerk SDK integration into existing backend
- New authentication endpoints alongside existing ones
- Enhanced user_profile table (add columns, not rebuild)
- Webhook handler components
- Organization/role mapping logic

This is an ENHANCEMENT project, not a greenfield build. We will integrate Clerk
authentication while preserving all existing functionality and infrastructure.

KEY BENEFITS:
- Centralized authentication across multiple domains
- Single source of truth for user authentication
- Unified role and permission management
- Support for social login providers
- Multi-tenant architecture support
- Reduced frontend complexity
- Enhanced security and compliance

SCOPE:
- Integrate Clerk SDK into EXISTING Spring Boot backend
- Create RESTful APIs for authentication operations
- Support JWT token validation and refresh
- ENHANCE existing multi-tenant user management
- Support social login providers (Google, GitHub, Facebook, Apple, etc.)
- Integrate Organization roles from Clerk
- Webhook support for user synchronization
- Comprehensive error handling and logging

OUT OF SCOPE (Phase 1):
- Migration of existing frontend authentication code
- Custom UI components for authentication
- Mobile app SDKs (future phase)
- Two-factor authentication customization (use Clerk default)
- Rebuilding existing infrastructure
- Changing current database schema structure

2. CURRENT STATE ANALYSIS
================================================================================

2.1 CURRENT AUTHENTICATION ARCHITECTURE
---------------------------------------
BACKEND (Spring Boot):
- Custom JWT authentication using JHipster security
- In-memory user details manager
- JWT credentials: username/password
- OAuth2 resource server with JWT decoder
- Base64-encoded secret key
- Token validity: 24 hours (86400 seconds)
- Remember-me token: 30 days (2592000 seconds)

FRONTEND (Next.js - Multiple Domains):
- Each domain has separate Clerk integration
- Domain-specific Clerk publishable keys
- Domain-specific Clerk secret keys
- Frontend-managed authentication state
- Direct Clerk API calls from browser
- No centralized user management

DATABASE (PostgreSQL):
- Table: user_profile
- Key Fields:
  * id (Long) - Primary key
  * tenant_id (String) - Tenant identifier
  * user_id (String) - Clerk user ID
  * email (String) - User email
  * first_name, last_name (String)
  * user_role (String) - User role
  * user_status (String) - Account status
  * profile_image_url (String)
  * phone, address fields
  * timestamps (created_at, updated_at)

UNIQUE CONSTRAINT:
- user_id is unique (but should be email + tenant_id for multi-tenant)

CURRENT SECURITY CONFIGURATION:
- SecurityFilterChain with OAuth2 JWT
- CorsConfiguration permitting all origins in dev
- InMemoryUserDetailsManager for API authentication
- Swagger UI publicly accessible

2.2 CURRENT LIMITATIONS
-----------------------
1. Multiple Clerk accounts required (one per domain)
2. Increased operational cost (multiple subscriptions)
3. Inconsistent user experience across domains
4. Difficult to manage roles and permissions
5. User data scattered across multiple Clerk instances
6. Complex frontend authentication logic
7. No centralized audit trail
8. Difficult to implement cross-domain features

3. PROBLEM STATEMENT
================================================================================

3.1 PRIMARY PROBLEM
-------------------
Currently, each frontend domain requires separate Clerk configuration and
credentials, leading to:
- Increased costs (multiple Clerk subscriptions)
- Management overhead (multiple configurations)
- User fragmentation (users can't move between domains seamlessly)
- Inconsistent authentication flows
- Difficult role management across domains

3.2 BUSINESS IMPACT
-------------------
- Higher operational costs ($25-99/month per domain)
- Slower feature development (implement auth in each frontend)
- Poor user experience (re-authentication across domains)
- Security risks (inconsistent security implementations)
- Compliance challenges (multiple audit trails)

3.3 TECHNICAL CHALLENGES
------------------------
1. Multi-tenant user identification (email + tenant_id uniqueness)
2. Token validation and refresh across domains
3. Role synchronization from Clerk Organizations
4. Social login provider configuration
5. Session management across multiple frontends
6. Webhook event handling for user updates
7. Migration of existing users from frontend-managed Clerk

5.2 COMPONENT DESCRIPTIONS
---------------------------

A. CLERK INTEGRATION SERVICE
   Purpose: Centralized service for all Clerk API interactions
   Responsibilities:
   - Initialize Clerk SDK with credentials
   - Validate JWT tokens from frontend
   - Create/update users in Clerk
   - Fetch user details from Clerk
   - Manage user sessions
   - Handle organization membership
   - Error handling and retry logic

B. AUTHENTICATION CONTROLLER
   Purpose: Expose RESTful APIs for frontend authentication
   Responsibilities:
   - Handle sign-up requests
   - Handle sign-in requests (email/password & social)
   - Token validation and refresh
   - User profile retrieval
   - Sign-out/session termination
   - Rate limiting and security

C. WEBHOOK HANDLER
   Purpose: Receive and process Clerk webhook events
   Responsibilities:
   - Verify webhook signatures
   - Process user lifecycle events
   - Synchronize user data with database
   - Handle organization membership changes
   - Audit trail logging
   - Idempotency handling

D. MULTI-TENANT USER SERVICE
   Purpose: Manage user profiles with tenant isolation
   Responsibilities:
   - Create/update user profiles
   - Enforce email + tenant_id uniqueness
   - Role assignment and validation
   - User search and filtering by tenant
   - Profile enrichment

E. SECURITY FILTER CHAIN
   Purpose: Secure API endpoints with Clerk token validation
   Responsibilities:
   - Extract Clerk JWT from Authorization header
   - Validate token with Clerk
   - Load user context and permissions
   - Enforce role-based access control
   - Handle authentication failures

6.2 NEW TABLE: clerk_user_tenant
---------------------------------
-- Junction table to support users belonging to multiple tenants

CREATE TABLE IF NOT EXISTS clerk_user_tenant (
    id BIGSERIAL PRIMARY KEY,
    user_profile_id BIGINT NOT NULL REFERENCES user_profile(id) ON DELETE CASCADE,
    tenant_id VARCHAR(255) NOT NULL,
    role VARCHAR(100), -- Tenant-specific role
    status VARCHAR(50) DEFAULT 'ACTIVE', -- ACTIVE, SUSPENDED, PENDING
    joined_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
    CONSTRAINT uq_clerk_user_tenant UNIQUE (user_profile_id, tenant_id)
);

CREATE INDEX idx_clerk_user_tenant_user ON clerk_user_tenant(user_profile_id);
CREATE INDEX idx_clerk_user_tenant_tenant ON clerk_user_tenant(tenant_id);
CREATE INDEX idx_clerk_user_tenant_status ON clerk_user_tenant(status);

COMMENT ON TABLE clerk_user_tenant IS 'Junction table for multi-tenant user relationships';
COMMENT ON COLUMN clerk_user_tenant.role IS 'Tenant-specific role (may differ from global user_role)';

6.3 NEW TABLE: clerk_organization_role
---------------------------------------
-- Map Clerk Organization roles to application roles

CREATE TABLE IF NOT EXISTS clerk_organization_role (
    id BIGSERIAL PRIMARY KEY,
    clerk_org_id VARCHAR(255) NOT NULL,
    clerk_role_name VARCHAR(100) NOT NULL, -- From Clerk Organizations
    application_role VARCHAR(100) NOT NULL, -- Mapped application role
    permissions JSONB, -- Role permissions as JSON
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
    CONSTRAINT uq_clerk_org_role UNIQUE (clerk_org_id, clerk_role_name)
);

CREATE INDEX idx_clerk_org_role_org ON clerk_organization_role(clerk_org_id);
CREATE INDEX idx_clerk_org_role_app_role ON clerk_organization_role(application_role);

COMMENT ON TABLE clerk_organization_role IS 'Maps Clerk Organization roles to application roles';

-- Example role mappings
INSERT INTO clerk_organization_role (clerk_org_id, clerk_role_name, application_role, permissions)
VALUES
('org_default', 'org:admin', 'ROLE_ORG_ADMIN', '{"canManageUsers": true, "canManageEvents": true, "canViewReports": true}'),
('org_default', 'org:member', 'ROLE_ORG_MEMBER', '{"canRegisterEvents": true, "canViewEvents": true}'),
('org_default', 'org:viewer', 'ROLE_VIEWER', '{"canViewEvents": true}')
ON CONFLICT (clerk_org_id, clerk_role_name) DO NOTHING;

6.4 NEW TABLE: clerk_webhook_event
-----------------------------------
-- Audit trail for all Clerk webhook events

CREATE TABLE IF NOT EXISTS clerk_webhook_event (
    id BIGSERIAL PRIMARY KEY,
    event_id VARCHAR(255) UNIQUE NOT NULL, -- Clerk event ID
    event_type VARCHAR(100) NOT NULL, -- user.created, user.updated, etc.
    clerk_user_id VARCHAR(255),
    payload JSONB NOT NULL, -- Full webhook payload
    processed BOOLEAN DEFAULT FALSE,
    processed_at TIMESTAMP,
    error_message TEXT,
    retry_count INTEGER DEFAULT 0,
    received_at TIMESTAMP NOT NULL DEFAULT NOW(),
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_clerk_webhook_event_type ON clerk_webhook_event(event_type);
CREATE INDEX idx_clerk_webhook_event_user ON clerk_webhook_event(clerk_user_id);
CREATE INDEX idx_clerk_webhook_processed ON clerk_webhook_event(processed);
CREATE INDEX idx_clerk_webhook_received_at ON clerk_webhook_event(received_at DESC);

COMMENT ON TABLE clerk_webhook_event IS 'Audit trail and processing queue for Clerk webhooks';

6.5 NEW TABLE: clerk_session
-----------------------------
-- Track active user sessions from Clerk

CREATE TABLE IF NOT EXISTS clerk_session (
    id BIGSERIAL PRIMARY KEY,
    session_id VARCHAR(255) UNIQUE NOT NULL, -- Clerk session ID
    clerk_user_id VARCHAR(255) NOT NULL,
    user_profile_id BIGINT REFERENCES user_profile(id) ON DELETE CASCADE,
    tenant_id VARCHAR(255),
    client_id VARCHAR(255), -- Device/client identifier
    ip_address VARCHAR(45),
    user_agent TEXT,
    status VARCHAR(50) DEFAULT 'ACTIVE', -- ACTIVE, EXPIRED, REVOKED
    expires_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    last_active_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_clerk_session_user ON clerk_session(clerk_user_id);
CREATE INDEX idx_clerk_session_profile ON clerk_session(user_profile_id);
CREATE INDEX idx_clerk_session_tenant ON clerk_session(tenant_id);
CREATE INDEX idx_clerk_session_status ON clerk_session(status);
CREATE INDEX idx_clerk_session_expires_at ON clerk_session(expires_at);

COMMENT ON TABLE clerk_session IS 'Tracks active user sessions from Clerk';

10. ROLE AND PERMISSIONS MANAGEMENT
================================================================================

10.1 ROLE HIERARCHY
-------------------
SYSTEM_ADMIN > ORG_ADMIN > ORG_MANAGER > ORG_MEMBER > USER > VIEWER

ROLE DEFINITIONS:
1. ROLE_SYSTEM_ADMIN
   - Full system access
   - Manage all tenants
   - View all data
   - System configuration

2. ROLE_ORG_ADMIN (from Clerk Organization: org:admin)
   - Manage organization members
   - Manage events and content
   - View reports and analytics
   - Configure organization settings

3. ROLE_ORG_MANAGER (from Clerk Organization: org:manager)
   - Create and manage events
   - Approve user registrations
   - View team reports
   - Manage content

4. ROLE_ORG_MEMBER (from Clerk Organization: org:member)
   - Register for events
   - View organization events
   - Update own profile
   - Submit requests

5. ROLE_USER (default)
   - View public events
   - Register for events
   - Update own profile
   - Basic access

6. ROLE_VIEWER
   - Read-only access
   - View public content only
   - No write permissions

10.2 CLERK ORGANIZATION INTEGRATION
------------------------------------

CLERK ORGANIZATIONS:
- Organizations represent groups/companies
- Each organization has members
- Members have organization-specific roles
- Roles: org:admin, org:member, org:viewer (customizable)

MAPPING PROCESS:

2. Backend fetches user's organizations and roles
3. Roles mapped using clerk_organization_role table
4. Application permissions applied based on mapped role

12.2 WEBHOOK SECURITY
----------------------
SIGNATURE VERIFICATION:
1. Clerk sends X-Clerk-Signature header
2. Backend verifies signature using webhook secret
3. Reject requests with invalid signature

IMPLEMENTATION:
```java
public boolean verifyWebhookSignature(
    String payload,
    String signature,
    String secret
) {
    try {
        String expectedSignature = computeHmacSha256(payload, secret);
        return MessageDigest.isEqual(
            signature.getBytes(),
            expectedSignature.getBytes()
        );
    } catch (Exception e) {
        return false;
    }
}
```

12.3 IDEMPOTENCY
----------------
PROBLEM: Webhooks may be delivered multiple times
SOLUTION: Track processed events by event_id

IMPLEMENTATION:

3. If exists, return 200 OK (already processed)

13. SECURITY CONSIDERATIONS
================================================================================

13.1 TOKEN SECURITY
-------------------
JWT TOKENS:
- Short-lived access tokens (1 hour)
- Long-lived refresh tokens (30 days)
- Secure storage on client (httpOnly cookies preferred)
- Token rotation on refresh
- Revoke tokens on sign-out

VALIDATION:
- Verify token signature with Clerk public key
- Check token expiration
- Validate issuer and audience claims
- Check user status (not suspended/banned)
- Rate limit token validation endpoints

13.2 API SECURITY
-----------------
RATE LIMITING:
- Authentication endpoints: 5 requests/minute per IP
- API endpoints: 100 requests/minute per user
- Webhook endpoint: 1000 requests/minute

CORS CONFIGURATION:
- Allow specific origins only (no wildcard in production)
- Allow credentials for cookie-based auth
- Restrict HTTP methods
- Set max age for preflight caching

INPUT VALIDATION:
- Validate all request parameters
- Sanitize user inputs
- Maximum length checks
- Email format validation
- Password complexity requirements

SQL INJECTION PREVENTION:
- Use prepared statements
- Never concatenate SQL queries
- ORM/JPA query builders
- Input sanitization

13.3 DATA SECURITY
------------------
ENCRYPTION AT REST:
- PostgreSQL database encryption
- Encrypt sensitive fields (SSN, credit cards)
- Use AWS KMS for key management

ENCRYPTION IN TRANSIT:
- HTTPS/TLS 1.3 only
- Strong cipher suites
- Certificate pinning
- HSTS headers

SENSITIVE DATA:
- Never log passwords
- Mask sensitive data in logs
- PII data encryption
- Secure credential storage (AWS Secrets Manager)

13.4 COMPLIANCE
---------------
GDPR:
- User consent management
- Right to be forgotten (user deletion)
- Data export capability
- Privacy policy

SOC 2:
- Audit logging
- Access controls
- Incident response
- Regular security audits

HIPAA (if applicable):
- PHI encryption
- Access logging
- BAA with Clerk
- Regular risk assessments

EXAMPLE:
```java
@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
@Testcontainers
class AuthenticationIntegrationTest {

    @Container
    static PostgreSQLContainer<?> postgres =
        new PostgreSQLContainer<>("postgres:16");

    @Test
    void testSignUpAndSignIn() {
        // Sign up
        SignUpRequest signUp = new SignUpRequest();
        signUp.setEmail("test@example.com");

        ResponseEntity<SignUpResponse> signUpResponse =
            restTemplate.postForEntity("/api/auth/sign-up", signUp, SignUpResponse.class);

        assertEquals(HttpStatus.CREATED, signUpResponse.getStatusCode());

        // Sign in
        SignInRequest signIn = new SignInRequest();
        signIn.setEmail("test@example.com");

        ResponseEntity<SignInResponse> signInResponse =
            restTemplate.postForEntity("/api/auth/sign-in", signIn, SignInResponse.class);

        assertEquals(HttpStatus.OK, signInResponse.getStatusCode());
        assertNotNull(signInResponse.getBody().getAccessToken());
    }
}
```
